## What I do

Help build the two scripts that handle the GitHub PR operations:

1. `create-pr.ts` — writes HTML files to `services/`, commits, pushes, creates/updates a PR in company-docs
2. `comment.ts` — posts or updates a comment on the source PR with a link to the docs PR and breaking change warnings

## Branch naming

Each source PR gets a dedicated branch: `docs/{service-name}/pr-{number}`

Examples: `docs/loan-api/pr-482`, `docs/web-checkout/pr-115`

## create-pr.ts flow

```
1. Configure git (user: docs-sync-bot)
2. Create or reset branch: docs/{service}/pr-{number}
3. For each DocFragment:
   - mkdir -p services/{fragment.outputPath directory}
   - Write fragment.html to services/{fragment.outputPath}
4. Write _meta-pr-{number}.json with fragment metadata
5. git add services/ && git commit
6. Force-push branch to company-docs remote
7. Check if PR already exists for this branch
   - Yes: log "updated", get existing PR URL
   - No: create new PR with title + body
8. Export DOCS_PR_URL for the comment step
```

## PR body format

```markdown
## Auto-Generated Documentation

**Source PR:** {link}
**Service:** {name}
**Generated:** {timestamp}

### Breaking Changes    (only if any exist)
- {severity icon} **{type}**: {entity}.{field} - {details}

### Documentation Files
- `services/loan-api/api/post-loan-apply.html` (api)
- `services/loan-api/models/LoanApplicationDto.html` (model)

---
*This PR was auto-generated by the documentation sync system.*
*It will be auto-merged when the source PR is merged.*
```

PR title: `docs({service}): PR #{number}` or `[BREAKING] docs({service}): PR #{number}`

## comment.ts flow

```
1. Build comment body with docs PR link + breaking changes
2. Search for existing comment with marker: <!-- docs-sync-comment -->
3. If found: update it (PATCH /comments/{id})
4. If not found: create new (POST /issues/{number}/comments)
```

The marker ensures the bot updates its own comment instead of creating duplicates.

## File output structure

```
services/{service}/
├── api/{method}-{path-slug}.html
├── models/{ModelName}.html
├── events/{EventName}.html
├── flows/{FlowName}.html
├── breaking-changes/pr-{number}.html
└── _meta-pr-{number}.json
```

Path slug: `POST /loan/apply` -> `post-loan-apply`

## GitHub API calls

```typescript
// Create PR
POST /repos/{owner}/{repo}/pulls
{ title, body, head: branch, base: "main" }

// Find existing PR
GET /repos/{owner}/{repo}/pulls?head={owner}:{branch}&state=open

// Create comment
POST /repos/{owner}/{repo}/issues/{number}/comments
{ body }

// Update comment
PATCH /repos/{owner}/{repo}/issues/comments/{id}
{ body }
```

All calls use `Authorization: Bearer {DOCS_REPO_TOKEN}` and `Accept: application/vnd.github+json`.

## Key rules

- Force-push to the docs branch on updates (idempotent — regenerates entirely)
- Never create duplicate PRs — always check for existing PR on the branch first
- Never create duplicate comments — use the HTML marker to find and update
- Breaking changes in PR title: `[BREAKING]` prefix
- Metadata JSON file written alongside HTML for the Astro site to consume
- The git user is `docs-sync-bot` / `docs-sync-bot@noreply.github.com`
- Commit message format: `docs({service}): auto-generated from PR #{number}`

## When to use me

Use this skill when building or debugging `create-pr.ts`, `comment.ts`, or any logic related to creating documentation PRs and posting comments on source PRs.
